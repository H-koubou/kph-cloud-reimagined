## 0. エグゼクティブサマリ

- ご提供いただいたログイン情報をもとに、**現行システム（山本建材＝本番／ダイイチ企業＝テスト）を読み取り専用で全画面調査**しました（データ変更は一切なし）。
- **重要な発見：両社は同一ベンダーの汎用「環境プラットホーム」を会社ごとに複製したインスタンス**でした（画面構成・URL設計・認証が完全共通、差分は社名と配色のみ）。
  → リプレイスも **「1つの優れたシステムを作り、会社ごとに展開する」マルチテナント設計**が有効。移行も共通設計で組めます。
- 現行は **約32画面**の堅実な業務システム。**業者→契約→現場→売上→請求→入金**の一本道がしっかり作られています。一方で、**ダッシュボード不在・JWNET電子マニフェスト未連携・各種手入力・地図/最適化の不在・予約者の可視性不足**といった、はっきりした伸びしろがあります。
- リプレイスは **①現状の全機能を補完（パリティ）→ ②ダッシュボード/JWNET/インボイス対応 → ③AIエージェント3本**の順で、土台を固めてから魔法を載せます。

## 1. 現状システムの全体像

### 1-1. テナント構成
| 環境 | 位置づけ |
|---|---|
| 山本建材（本番） | 実運用中の本番。今回は閲覧のみ |
| ダイイチ企業（テスト） | 同プラットフォームで作られた実例。深掘り調査はこちら |

→ **同一基盤のマルチテナント**。リプレイス後も「社ごとにテナント展開」で横展開可能。

### 1-2. 技術・構成
- サーバーサイドレンダリングの業務システム（PHPフレームワーク）。Bootstrap系UI。
- ロールは3種：**管理者 / ドライバー / 予約者**。共通ログイン後、**ロールごとにメニューを出し分け**。
- 予約者だけは専用の入口（排出事業者＝取引先が自分でWeb予約する窓口）。

### 1-3. データモデル相関（ここが設計の肝）

```
業者(client) ── 全マスターの起点（排出/運搬/処分を「区分」で混在管理）
  │   業者レベルで保持：締日・支払条件・請求書発行要否・インボイス番号
  ├─→ 許可証(permit)        ：許可種類6種・有効期限・許可品目・PDF
  ├─→ 銀行(account)         ：自社振込先（メイン口座→請求書へ転記）
  ├─→ 請求先(billing)       ：1業者に複数請求先（締日・メイン指定）
  ├─→ 積替保管施設(facility)・処分場(disposal)：許可番号・上限㎥
  │
  └─→ 契約(contract) ★中核ハブ
         ├ 排出事業者 / 収集運搬会社 / 処分会社 / 積替保管 / 請求先 を全て紐付け
         ├ 請求区分【請求先ごと / 契約ごと / 現場ごと】★請求書の集計単位を決定
         ├ 許可番号（排出・搬入・処分の場所ごと）
         └─→ 単価(price)：業者×契約×現場×品目×単位、適用期間、運搬/処分/買取/販売価格
                 └─→ 現場(place)：契約に紐付き。ドライバー/車種→配車・日報、請求先→現場別請求
```

### 1-4. 業務トランザクション・フロー
```
予約(持込/回収) → 配車ボード → 現場作業[受付/計量/搬出で品目・重量・マニフェスト番号入力]
   → 売上(自動生成) ← 単価マスター
   → 請求書(売上を 業者計/現場計/売上計 で集約。集計単位＝契約の請求区分)
   → 入金(複数請求へ消込) → 銀行口座
法令台帳：マニフェスト(紙7票・A〜E票返却日) ／ 日報(アルコールチェック＋走行メーター) ／ 許可証(有効期限)
```

## 3. 現状の弱点・欠落（＝リプレイスの伸びしろ）

**A. 連携・自動化（最優先）**
1. **JWNET電子マニフェストAPI連携** — 現状は紙台帳＋番号手入力で電子は別管理（二重管理）。**最重要**。
2. **計量機（トラックスケール）自動取込** — 重量が手入力。
3. **アルコール検知器の自動取込** — 検知値が手入力（点呼記録の信頼性）。
4. **会計連携／CSV** — 売上・入金・請求の入出力、銀行入金データでの自動消込。

**B. 税制・請求**
5. **インボイス制度・複数税率対応**（軽減8%欄・税率別内訳・適格番号の帳票反映）。
6. **入金消込の可視化**（請求残額・未収一覧・自動消込）。

**C. 期限・アラート・ダッシュボード**
7. **経営ダッシュボード新設**（未処理予約/未請求/未入金/許可証失効/契約満了をKPI・アラート化）。
8. **許可証失効アラート**（契約・施設・処分場が許可番号を参照するため必須）。
9. **契約満了アラート**。

**D. マスター・UX統制**
10. **権限のロール化**（テンプレート/グループ）。
11. **業者の360°ビュー**（許可証/銀行/契約/現場/施設をタブ統合）。
12. **現場名の名寄せ統合**／**業者の役割（排出/運搬/処分）整理**。
13. **巨大フォームの分割・ウィザード化**（特に持込受付・契約）。
14. **コンテナ個体トラッキング**／**施設容量の実績突合**。

## 5. AIエージェント3本（ご希望）の設計

### ① 配車AIエージェント
- **置き換える現状**：午前/午後マトリクスへの**手動割当**、地図は外部リンクのみ。
- **入力**：当日の持込・回収予約／車輌キャパ（2㎥・8㎥コンテナ・ユニック等）／時間指定（何時以降・何時〜何時）／**現場座標（既存の座標データを活用）**。
- **出力**：巡回順とドライバー割当を**AIが提案**→人がワンタップ承認（採用/却下）。地図上に最適ルートを描画、臨時回収はドライバーへ一斉通知。
- **効果**：配車時間の大幅短縮。既存の「回収希望日・時間指定」データがそのまま学習素材。

### ② ドライバーアプリ（撮るだけ）
- **置き換える現状**：PC管理画面の制限ビュー流用、日報はアルコール＋メーターのみ。
- **機能**：現場写真→**品目・数量・不適物をAI判定**／メーター・アルコール・**車両点検チェックを撮影・音声で起票**／電子サイン／オフライン対応。
- **効果**：現状の手入力日報・受付重量・マニフェスト品目入力を自動化。現場が止まらない。

### ③ 喋るだけ事務（＋予約者AIエージェント）
- **置き換える現状**：契約書作成依頼の長大フォーム、持込/回収予約の手入力、品目が無い時の「担当者へ連絡」＝電話運用。取消は申請→承認の人手。
- **機能**：電話・チャットの**自然言語から予約・契約依頼を自動入力**／品目が選択肢に無い時はAIがマッピング提案／予約者からの**受入混雑・空き枠・進捗・マニフェスト・請求の問い合わせをAIが一次対応**。
- **効果**：事務の電話依存を削減、予約者の自己解決を促進。

## 7. 移行方針

- **同一プラットフォームのマルチテナント**のため、社ごとの**並行運用→段階移行**が可能。
- 既存の業者・契約・現場・単価・許可証・マニフェスト台帳・売上/請求/入金を**マッピングして移行**。
- 移行中もJWNET運用・請求業務が止まらないよう、Phase 1（パリティ）達成時点で切替判断。

## 8. ご確認いただきたいこと

1. **隠れ機能の実体**：「産廃数量管理」「収集運搬実績報告」は何の機能か（JWNET報告／行政の実績報告か）。リプレイス範囲に直結します。
2. **JWNET運用の現状**：今は電子マニフェストをJWNETのWebで別途運用されていますか？（連携設計の前提）
3. **計量機・アルコール検知器**：機種・型番（自動取込の可否判断）。
4. **会計ソフト**：現在お使いの会計/銀行データ連携の有無。
5. **税率**：軽減税率8%対象の取引はありますか（請求書の複数税率対応の要否）。
6. **リプレイス優先順位**：Phase 1（全補完）の中で特に急ぐ画面はどれか。
